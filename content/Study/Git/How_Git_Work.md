### 1. Git 구조적 한계

|항목|설명|
|---|---|
|**스냅샷 기반 저장**|Git은 변경 이력을 “diff”가 아니라 **전체 파일 스냅샷**으로 저장합니다. 즉, 100MB짜리 파일을 한 번 수정하면 새 커밋마다 100MB가 또 저장됩니다.|
|**레포지토리 크기 폭증**|한두 번 수정만 해도 `.git` 폴더 내부의 히스토리가 수 GB로 커집니다. 클론(clone), fetch, pull 시 전부 다운로드해야 함.|
|**분산 저장 특성**|Git은 중앙 서버형이 아니라 **모든 클라이언트가 전체 이력 복제**하는 구조라, 대용량 파일이 있으면 **모든 사용자가 전송 부담**을 가짐.|

→ 따라서 GitHub은 서버 부하와 사용자 네트워크 비용을 줄이기 위해 **100MB 이상 파일에 경고 또는 거부(error)** 정책을 둡니다.

### ⚙️ 2. GitHub 서버 및 성능 이슈

|이유|상세 설명|
|---|---|
|**스토리지 관리 효율**|GitHub은 수억 개의 repo를 운영하므로, 대용량 바이너리 파일 저장은 백엔드 I/O 및 백업 부담이 큼.|
|**네트워크 트래픽 제한**|대용량 push/pull이 빈번하면 Git 전송 프로토콜(HTTP/SSH) 과부하 유발.|
|**인덱싱 및 diff 불가**|`.pth`, `.zip`, `.png` 등 바이너리 파일은 Git의 diff/merge 기능을 쓸 수 없음. 즉, 버전 관리 가치가 적고 비용만 큼.|

---

### 💡 3. 대안: Git LFS (Large File Storage)

- Git LFS는 대용량 파일 본문을 외부 서버(S3 등)에 저장하고,  
    Git에는 **“포인터 파일(메타데이터)”**만 남겨둡니다.
    
- 즉, Git의 버전 관리 기능은 유지하면서 **대용량 파일은 별도 저장소에서 관리**함.
    
- GitHub이 공식적으로 LFS를 권장하는 이유가 바로 이것입니다.

### ✅ 요약 (3줄)

- Git은 스냅샷 기반이라 대용량 파일이 있으면 커밋마다 중복 저장됨.
    
- GitHub은 서버 부하·트래픽·저장 효율을 위해 100MB 이상 push를 차단.
    
- 해결책은 Git LFS 사용 또는 대용량 파일을 외부 저장소(Drive, S3 등)로 분리.
